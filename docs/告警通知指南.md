# 告警通知指南

> 本文档介绍告警规则的配置方式、四种告警类型以及三种通知渠道的使用方法。

---

## 一、告警规则

### 1.1 字段说明

| 字段 | 说明 |
|------|------|
| `name` | 规则名称，如"错误率告警-订单接口" |
| `description` | 规则描述 |
| `alertType` | 告警类型：`ERROR_RATE` / `LATENCY` / `RATE_LIMIT` / `CONSECUTIVE_FAIL` |
| `apiCode` | 监控的接口编码（为空则监控所有接口） |
| `threshold` | 触发阈值（含义随 alertType 而变） |
| `windowMinutes` | 统计时间窗口（分钟），默认 5 分钟 |
| `channelList` | 通知渠道数组：`["EMAIL", "DINGTALK", "WECOM"]` |
| `receivers` | 接收人列表（邮箱地址 / 钉钉 Webhook URL 等） |
| `enabled` | 是否启用 |

### 1.2 四种告警类型

#### ERROR_RATE（错误率告警）

当接口在统计窗口内的错误率超过阈值时触发。

- **threshold**：错误率百分比，如 `5.0` 表示错误率超过 5% 时告警
- **windowMinutes**：统计时间窗口，默认为 5 分钟
- **错误定义**：HTTP 状态码 ≥ 400 视为失败

```json
{
  "name": "订单接口错误率告警",
  "alertType": "ERROR_RATE",
  "apiCode": "ORDER_API",
  "threshold": 5.0,
  "windowMinutes": 5,
  "channelList": ["EMAIL"],
  "receivers": ["ops@example.com"]
}
```

#### LATENCY（延迟告警）

当接口在统计窗口内的平均响应时间超过阈值时触发。

- **threshold**：平均响应时间（毫秒），如 `3000` 表示平均响应超过 3 秒时告警
- **windowMinutes**：统计时间窗口，默认为 5 分钟

```json
{
  "name": "支付接口延迟告警",
  "alertType": "LATENCY",
  "apiCode": "PAY_API",
  "threshold": 3000,
  "windowMinutes": 5,
  "channelList": ["DINGTALK"],
  "receivers": ["https://oapi.dingtalk.com/robot/send?access_token=xxx"]
}
```

#### RATE_LIMIT（限流告警）

当接口在统计窗口内的请求量超过阈值时触发。

- **threshold**：请求量上限，如 `1000` 表示 5 分钟内请求超过 1000 次时告警
- **windowMinutes**：统计时间窗口，默认为 5 分钟

```json
{
  "name": "高频调用告警",
  "alertType": "RATE_LIMIT",
  "threshold": 1000,
  "windowMinutes": 5,
  "channelList": ["WECOM"],
  "receivers": ["https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx"]
}
```

#### CONSECUTIVE_FAIL（连续失败告警）

当接口连续失败次数超过阈值时触发（立即告警，不等待时间窗口）。

- **threshold**：连续失败次数，如 `3` 表示连续 3 次失败即告警
- **windowMinutes**：不使用（实时检测）

```json
{
  "name": "服务连续失败告警",
  "alertType": "CONSECUTIVE_FAIL",
  "apiCode": "AUTH_API",
  "threshold": 3,
  "channelList": ["EMAIL", "DINGTALK"],
  "receivers": ["ops@example.com", "https://oapi.dingtalk.com/robot/send?access_token=xxx"]
}
```

## 二、通知渠道

### 2.1 支持的渠道

| 渠道 | 标识符 | 接收人格式 | 说明 |
|------|--------|-----------|------|
| 邮件 | `EMAIL` | 邮箱地址，多个用逗号分隔 | 通过 QQ 邮箱 SMTP 发送 |
| 钉钉 | `DINGTALK` | Webhook URL | 群机器人 Webhook |
| 企业微信 | `WECOM` | Webhook URL | 企业微信群机器人 |

### 2.2 渠道配置

#### 邮件配置（application.yml）

```yaml
spring.mail.host: smtp.qq.com
spring.mail.port: 587
spring.mail.username: your-email@qq.com
spring.mail.password: your-app-password
mail.from: your-email@qq.com
```

> ⚠️ QQ 邮箱需要开启 **SMTP 服务**并获取**授权码**（非登录密码）。

#### 钉钉配置

1. 群设置 → 智能群助手 → 添加机器人
2. 选择"自定义"机器人
3. 安全设置选择"自定义关键词"，填入如 `告警`
4. 复制 Webhook URL 填入接收人

#### 企业微信配置

1. 企业微信群 → 添加群机器人
2. 复制 Webhook URL 填入接收人

## 三、告警评估机制

### 3.1 定时扫描

`AlertEvaluationService` 每 **60 秒**执行一次评估，遍历所有启用的告警规则：

```
每60秒扫描所有告警规则
    ↓
判断 alertType 类型
    ↓
查询 log 库统计指标（ERROR_RATE/LATENCY/RATE_LIMIT）或调用记录（CONSECUTIVE_FAIL）
    ↓
与 threshold 比较
    ↓
超过阈值 → 检查冷却期 → 发送通知 → 写入告警记录
```

### 3.2 冷却机制

告警触发后进入冷却期（默认 30 分钟），冷却期内同一规则不重复告警。冷却时长可通过规则配置覆盖。

### 3.3 跨库查询

告警评估需要同时查询 **config 库**（规则配置）和 **log 库**（调用记录），通过显式指定 `EntityManager` 实现跨库查询。

## 四、告警记录

每次告警触发后生成一条告警记录，包含：

| 字段 | 说明 |
|------|------|
| `ruleId` | 关联的规则 ID |
| `ruleName` | 规则名称（冗余存储） |
| `alertType` | 告警类型 |
| `triggerValue` | 触发时的实际值（如实际错误率 8.5%） |
| `threshold` | 阈值 |
| `status` | 状态：`TRIGGERED` / `ACKNOWLEDGED` / `RESOLVED` |
| `triggeredAt` | 触发时间 |
| `acknowledgedAt` | 确认时间 |
| `acknowledgedBy` | 确认人 |
| `resolvedAt` | 解决时间 |

## 五、API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/alert/rules` | GET | 查询告警规则列表 |
| `/api/alert/rules` | POST | 新建告警规则 |
| `/api/alert/rules/{id}` | PUT | 更新告警规则 |
| `/api/alert/rules/{id}` | DELETE | 删除告警规则 |
| `/api/alert/rules/{id}/toggle` | PUT | 启用/停用规则 |
| `/api/alert/rules/{id}/test` | POST | 测试规则（模拟触发，不发真实通知） |
| `/api/alert/records` | GET | 查询告警记录列表 |
| `/api/alert/records/{id}/ack` | PUT | 确认告警 |
| `/api/alert/records/{id}/resolve` | PUT | 解决告警 |

## 六、常见问题

**Q：告警没有触发？**
1. 确认规则 `enabled` 为 `true`
2. 确认接口有实际调用记录（log 库有数据）
3. 检查日志是否有 `AlertEvaluationService` 的定时任务执行记录

**Q：邮件没收到？**
1. 检查 `spring.mail` 配置是否正确
2. 确认 `mail.from` 与 `spring.mail.username` 一致
3. 检查垃圾邮件箱

**Q：钉钉/企微没收到？**
1. 确认 Webhook URL 正确且机器人未过期
2. 确认消息体格式符合平台要求（JSON 格式）
